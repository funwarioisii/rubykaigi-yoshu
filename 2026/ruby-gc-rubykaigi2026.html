Section 01
Ruby GC の進化
Ruby 1.8
Mark & Sweep（STW） — 単純だがすべてのスレッドを停止
Ruby 2.0
Bitmap Marking — CoW フレンドリーに。FL_MARK ビットの空きが後の WB に繋がる
Ruby 2.1
RGenGC（世代別 GC） — Write Barrier 導入。WB-protected / unprotected の概念を発明
Ruby 2.2
Incremental GC（RincGC） — Major GC を分割実行。WB を使い三色マーキングの整合性を保証
Ruby 2.7+
GC Compaction — オブジェクトを移動してフラグメンテーションを解消
Ruby 3.4
Modular GC — GC を共有ライブラリとして分離。MMTk 統合を同梱（実験的）
Ruby 4.x（進行中）
Next-Gen GC — MMTk が Default GC と同等性能に到達。Moving Immix、Ractor-Local GC の研究
Section 02
Write Barrier — なぜ必要か
OLD Object obj A
→
YOUNG Object obj F
⚠️
Write Barrier 参照を検出して
Remembered Set に記録

WB-Protected

参照変更がすべて GC に通知される。Old 世代への昇格が可能。String, Array, Hash など主要クラスが該当。

Minor GC でスキップ可能

WB-Unprotected

C 拡張との互換性のため WB を挿入できないオブジェクト。Old に昇格不可。Minor GC のたびにスキャンが必要。

互換性のための妥協

Ruby の独自性

「全オブジェクトに WB を要求しない」段階的導入戦略。これにより既存の C 拡張を壊さず世代別 GC を実現した。

Gradual WB Insertion
Section 03
Remembered Set の記録方式

Write Barrier が old→young 参照を検出した後、どう記録するかの 3 つの戦略。

🎯

Object ベース

粒度: オブジェクト単位

参照を持つオブジェクト自体をセットに登録。精度最高だが WB コスト・メモリが大きい。
Ruby の現行方式。

スキャン精度 ████████▒
🃏

Card ベース

粒度: ~512 バイト

ヒープを小領域（カード）に分割し dirty ビットを立てるだけ。WB が極めて軽い。
JVM HotSpot (G1GC) の方式。

スキャン精度 █████▒▒▒▒
📄

Page ベース

粒度: ~4KB（ページ）

OS の mprotect で書き込み禁止にし、page fault で検出。アプリ側の WB コード不要だが trap が重い。

スキャン精度 ██▒▒▒▒▒▒▒
Section 04
Modular GC × Ractor — 未来の交差点
Ruby 4.x のメモリ管理
Modular GC
gc_impl.h API で GC を差し替え可能に。dlopen で共有ライブラリをロード
MMTk
汎用 GC フレームワーク。Moving Immix / StickyImmix 等のアルゴリズムを提供
Ractor
Actor ベースの並列実行。各 Ractor に独自 GVL。オブジェクト空間の分離が課題
Write Barrier
世代別・インクリメンタル GC の正しさの根幹。C 拡張との互換性が制約に
Ractor-Local GC
Ractor ごとにヒープを分離し独立 GC。STW の影響範囲を限定する将来構想
C 拡張の近代化
WB 対応・Compaction 対応の新 API へ移行。旧 API はサイレントに性能を阻害
