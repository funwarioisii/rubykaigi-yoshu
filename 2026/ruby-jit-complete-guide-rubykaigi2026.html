Chapter 0 — 基礎

そもそも JIT とは何か

「実行しながらコンパイルする」という発想

プログラムの実行方式は大きく 3 つに分けられます。

インタプリタはコードを 1 行ずつ解釈して実行します。起動は速いですが、同じコードを何度も解釈し直すため繰り返し実行が遅い。AOT コンパイラ（C や Rust）は実行前にすべてをマシンコードに変換します。実行は速いですが、動的型付けの Ruby には適用しにくい——実行時まで型がわからないからです。

JIT（Just-In-Time）コンパイラは第三の道です。最初はインタプリタとして動き、「よく実行されるコード」を見つけたらそこだけマシンコードにコンパイルする。実行時に得た型情報を使って最適化できるのが最大の強みです。

3 つの実行方式
Interpreter 逐次解釈
起動速い・実行遅い
JIT 実行時コンパイル
いいとこ取り
AOT Compiler 事前コンパイル
起動遅い・実行速い
Ruby の歩み Ruby 1.8 以前は AST を直接たどるインタプリタ。Ruby 1.9 で ko1 が YARV（バイトコード VM）を導入し大幅に高速化。しかしそれでも Python や JavaScript に比べると遅い——そこで JIT の導入が模索されました。
Chapter 1 — ベンチマーク

optcarrot — Ruby3x3 を測る「人参」

ファミコンが 60fps で動けば Ruby は 3 倍速くなった証拠

Matz が「Ruby 3 は Ruby 2 の 3 倍速くなる」（Ruby3x3）を宣言。この目標を測る「楽しい」ベンチマークとして遠藤侑介（@mame）さんが作ったのが optcarrot——純 Ruby で書かれた NES エミュレータです。

本物の NES は 60fps。Ruby 2.0 で約 20fps → 3 倍の 60fps に届けば Ruby3x3 達成。純粋に CPU 集約型で、GC や I/O の改善では速くならないため、実行エンジン自体の速度を正直に反映します。5000 行未満・外部依存ゼロで、全 Ruby 実装で動くのも強みです。

Chapter 2 — 第一世代

MJIT — Ruby 初の JIT（2.6〜）

バイトコードを C に変換し gcc でコンパイルする力技

Ruby 2.6 で導入された MJIT（Method-based JIT）は、YARV バイトコードを C ソースコードに変換し、外部の C コンパイラ（gcc / clang）でコンパイルするアプローチでした。

Ruby コード
↓
YARVバイトコード
↓
MJITC ソースコードを生成
↓
gcc / clangネイティブコード（.so）
↓
dlopen実行時にロード
MJIT の課題 外部コンパイラの起動が遅く、ウォームアップに時間がかかる。コンパイルが別プロセスで走るためメモリも食う。Ruby 特有の動的性質（メソッド再定義、オープンクラスなど）への対応も難しかった。Ruby 3.3 で RJIT に改名、Ruby 4.0 で削除されました。
Chapter 3 — 現在の主力

YJIT — Lazy Basic Block Versioning

「型を見てからコンパイル」で本番投入に耐える JIT

Shopify の Maxime Chevalier-Boisvert が中心となり Ruby 3.1 で導入。MJIT とは根本的に異なるアプローチを取ります。Rust で書かれ、外部コンパイラ不要。Ruby プロセス内でインクリメンタルにマシンコードを生成します。

YJIT の核心は Lazy Basic Block Versioning（LBBV）です。

LBBV の動作イメージ
① メソッドが呼ばれる。最初はインタプリタで実行し、型情報を観測
② 閾値を超えたら、実際に実行されたパスだけをコンパイル（lazy）
③ 観測した型に特化したマシンコードを生成
④ 別の型が来たら → その型用の別バージョンを追加生成（versioning）
⑤ 想定外の型 → インタプリタにフォールバック（side exit）

強み

ウォームアップが速い。コンパイルが軽い。side exit でいつでもインタプリタに戻れるので安全。Shopify, Discourse, Mastodon, GitHub 等で本番稼働。フラグ 1 つで 20〜30% 高速化。

限界

ベーシックブロック単位なのでメソッド全体を見渡す最適化（定数畳み込み、ループ最適化など）が困難。LBBV は独自手法で教科書に載っていないため、外部貢献者の参入障壁が高い。

Chapter 4 — 次世代

ZJIT — 教科書的アーキテクチャで天井を破る

SSA 形式の IR を導入した「本格派」コンパイラ

YJIT と同じ Shopify チームが開発。Ruby 4.0（2025 年末）に実験的に同梱。--zjit で有効化。YJIT の「性能の天井」と「貢献のしにくさ」を同時に解決するために、アーキテクチャを一新しています。

YJIT との最大の違いは HIR の存在です。YARV のスタックベース命令列を、SSA（Static Single Assignment）形式のグラフ IR に変換することで、コンパイラの教科書に載っている最適化手法を体系的に適用できます。

現状と今後 Ruby 4.0 時点でインタプリタより速いが YJIT ほどではない。目標は Ruby 4.1 で YJIT を超え本番投入可能にすること。HIR のおかげで今後は最適化パスの追加で継続的に性能が向上する伸びしろがあります。

Chapter 6 — 別ルート

ASTro — VM を迂回する実験

YARV の生みの親が YARV を使わない道を探る

ko1（笹田耕一）さんが開発中の ASTro は、YJIT/ZJIT とは全く異なるアプローチを取ります。YARV バイトコードを経由せず、AST から直接 C コードを生成し、gcc でコンパイルしてネイティブコードを得ます。

YJIT / ZJIT の道

YARV 経由

Ruby → AST → YARV バイトコード → JIT → ネイティブコード

YARV のスタックマシン命令を入力とするため、YARV の命令セット設計に制約される。

ASTro の道

YARV を迂回

Ruby → AST → ASTro で最適化 → C コード → gcc → ネイティブコード

AST にはソースの構造情報が豊富に残っている。YARV 変換で失われる情報を活用できる可能性。

なぜこれが面白いか ko1 さんは YARV を作った本人。VM の限界を最もよく知っている人が「VM を使わないアプローチ」を試しているという構図。研究段階であり本番提案ではないが、VM の暗黙の制約を言語化するという意味で、ZJIT チームにとっても示唆のある実験です。
Chapter 7 — 全体像

JIT と GC — Ruby 高速化の両輪

前回のインフォグラフィックと繋がるポイント

Ruby の高速化は JIT（実行速度）と GC（メモリ管理）の 2 本柱で進んでいます。しかもこの 2 つは独立ではなく、深く絡み合っています。

🔗

JIT と Write Barrier

JIT がコンパイルしたコードも Write Barrier を正しく呼ぶ必要がある。YJIT/ZJIT は WB のコストを下げるためにインライン化しており、WB の正しさはJIT コードの正しさに直結する。

⏸️

JIT と GC ポーズ

JIT で実行が速くなっても GC で止まれば意味がない。YJIT/ZJIT は TracePoint を検出するとコンパイル済みコードを破棄しインタプリタに戻す仕組みがあり、GC との協調が必須。

📦

JIT と GC Compaction

GC がオブジェクトを移動（compact）すると、JIT コード内のポインタが無効になる。JIT コードは GC の移動に追従する仕組みが必要で、これは Modular GC で GC を差し替えた場合にも問題になる。
