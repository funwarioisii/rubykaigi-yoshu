Chapter 0 — 前提

GVL — すべての出発点

なぜ Ruby の並行性は「複雑」なのか

CRuby には GVL（Global VM Lock）があります。1 プロセス内で同時に Ruby コードを実行できるスレッドは 1 つだけ。CPU コアが 8 個あっても、Ruby コードは 1 コアしか使えません。

ただし GVL は「Ruby コードの実行」だけをロックします。I/O 待ち（DB クエリ、HTTP 通信、ファイル読み書き）の間は GVL を手放す。これが Thread / Fiber / Ractor の使い分けの根拠になります。

GVL が制約するもの・しないもの
GVL がロック Ruby コード実行
計算、メソッド呼出、オブジェクト操作
GVL を手放す I/O 待ち
DB クエリ、HTTP、ファイル、sleep
Chapter 1

Thread — OS スレッド

I/O 待ちの間に他の仕事をする

Ruby 1.9 からの並行性プリミティブ。OS が管理するスレッドで、プリエンプティブ（OS が切替タイミングを決める）。GVL のため Ruby コードの並列実行はできないが、I/O 待ちの間は他のスレッドが動ける。

Puma（Rails のデフォルト Web サーバー）はこの仕組みで動いています。1 プロセスに複数スレッドを立て、あるリクエストが DB を待っている間に別のリクエストを処理する。

Thread の限界 1 スレッドあたり ~8MB のメモリ。OS のコンテキストスイッチもそこそこ重い。そして共有メモリなので Mutex によるロックが必要——デッドロックやレースコンディションのリスクが常にある。同時接続数が増えると、スレッド数 × メモリが線形に増大する。
Chapter 2

Fiber — 軽量な協調スケジューリング

1 スレッドで数千の同時接続

Fiber はスレッドの中で動く軽量なコルーチンです。1 つあたり ~4KB と極めて軽量。OS ではなくプログラマ（または Fiber Scheduler）が切替タイミングを決める（cooperative scheduling）。

Ruby 3.0 で Fiber Scheduler インターフェースが導入され、I/O ブロック時に自動的に別の Fiber に切り替える仕組みが言語レベルでサポートされました。これにより「1 スレッド・数千 Fiber」で大量の同時接続を捌けます。

Fiber の弱点 協調的なので、1 つの Fiber が CPU を長時間使うと他の Fiber が飢餓状態になる。また GVL の中で動くため、CPU バウンドな処理の並列化はできない。あくまで I/O 多重化に特化した仕組み。
Chapter 3

Ractor — 真の並列実行

GVL の壁を超える。ただし代償がある

Ruby 3.0 で導入（実験的）。各 Ractor が独自の GVL を持ち、Ruby コードを複数コアで本当に並列実行できます。メモリは Ractor 間で共有しない（メッセージパッシングで通信）。

Ractor の課題 DB コネクションプール、ロガー、グローバル設定などが共有できない。既存の gem のほとんどが Ractor 非対応。Rails アプリをそのまま Ractor 化するのは現実的でない。CPU バウンドな特定処理（デシリアライズ、画像処理、ML 推論など）をプールに切り出す使い方が現実的。Ruby 4.0 では Ractor::Port による通信の改善も入りました。

Chapter 6 — データ処理

Apache Arrow — Ruby をデータの世界に接続する

DuckDB、Athena、Parquet...すべてが Arrow で繋がる

Apache Arrow は列指向のインメモリデータフォーマットです。行指向（CSV、従来の DB 行）ではデータがバラバラに散らばりますが、Arrow では同じカラムのデータがメモリ上に連続して並ぶため、分析処理が桁違いに速い。現代のデータ処理ツールの共通言語です。

C++ バインディング経由

Arrow C++ → Arrow GLib → GObject Introspection → Ruby。超高速だがインストールが大変（C++ ビルド環境が必要）。大規模データ処理向け。

Pure Ruby Arrow（New）

RubyKaigi 2026 で発表

C 依存なし。gem install だけで入る。「データ交換」に特化し、高コストな処理は期待しない。より多くの gem が Arrow 入出力を気軽にサポートできるように。

DuckDB / Athena との繋がり DuckDB: インプロセス分析 DB。Arrow ネイティブ対応。Pure Ruby Arrow があれば Ruby アプリが DuckDB のクエリ結果を Arrow 形式で直接受け取れる。CSV 変換不要。
AWS Athena: S3 上の Parquet（Arrow ベース）をクエリするサーバーレスサービス。Ruby から Athena の結果を Arrow で受け取り、そのまま処理 → DuckDB にロードするパイプラインが Ruby だけで組める。
つまり Pure Ruby Arrow は「Ruby をデータ処理パイプラインの一級市民にする」鍵。
